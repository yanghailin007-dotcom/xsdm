# 第一阶段生成逻辑说明

## 概述

本文档说明了小说生成系统第一阶段的生成逻辑限制和实现细节。

## 第一阶段的范围

**第一阶段只生成到中级事件为止**，具体包括：

1. ✅ **重大事件** - 阶段的主龙骨
2. ✅ **中型事件** - 重大事件的分解
3. ✅ **特殊情感事件** - 情感点缀

**不在第一阶段生成**：

4. ❌ **章节事件** - 中型事件的进一步分解
5. ❌ **场景事件** - 章节事件或直接场景的分解

## 实现细节

### 1. 事件分解器 (EventDecomposer)

`decompose_major_event()` 方法：
- 只将重大事件分解为中型事件和特殊情感事件
- 返回的结构包含：
  - `composition`: 中型事件的"起承转合"结构
  - `special_emotional_events`: 特殊情感事件列表

**不调用**：
- `smart_decompose_medium_events()` - 这个方法用于第二阶段的进一步分解

### 2. 场景组装器 (SceneAssembler)

`assemble_final_plan()` 方法（第一阶段版）：
- 接收已分解的重大事件（包含中型事件）
- 收集特殊情感事件
- 构建 `chapter_scene_events` 为空列表 `[]`
- 添加 `generation_phase: "phase_one"` 标记

**关键修改**：
```python
"event_system": {
    "overall_approach": "第一阶段生成：包含重大事件、中型事件和特殊情感事件。章节事件和场景事件将在第二阶段生成。",
    "major_events": final_major_events,
    "special_emotional_events": all_special_events,
    "chapter_scene_events": []  # 第一阶段为空，第二阶段填充
}
```

### 3. 验证逻辑 (StagePlanManager)

`_validate_and_optimize_writing_plan()` 方法：
- 验证中型事件的章节覆盖
- 未覆盖的章节不会导致失败，只会发出警告
- 这些未覆盖章节将在第二阶段通过进一步分解来覆盖

## 第二阶段的工作

第二阶段将基于第一阶段的中型事件，进一步分解为场景。

### AI自由决策方案 ⭐

**核心理念**：让AI根据中型事件的具体情况自由决策最优结构，不强制规定章节数和场景数的对应关系。

#### AI需要决策的内容：

1. **是否需要分解为章节事件？**
   - 如果章节内容相对独立 → 分解为章节事件
   - 如果章节讲述连续情节 → 直接生成场景序列

2. **每章包含多少个场景？**
   - 根据需要灵活调整（3-8个场景都合理）
   - 重要转折章可以有更多场景
   - 过渡章可以有较少场景

3. **哪些场景可以合并？哪些需要扩展？**
   - 相关场景可以合并到一章内
   - 重要场景可以扩展成多个章节
   - 根据叙事需要自由决定

#### 决策考虑因素：

- **目标复杂度**：目标简单可以一章完成，复杂需要多章展开
- **情感弧线**：情绪需要铺垫和发展，不要急躁
- **节奏控制**：读者需要呼吸的空间，不要信息过载
- **叙事连贯**：场景之间要有自然的过渡和递进

#### 唯一约束：

**✅ 严禁在不同章节使用相同或相似的场景内容**
- 每个场景都应该有独特的进展和意义
- 章节之间要有明确的递进关系

#### 可能的输出结构：

**方案A**：不分解章节，直接生成场景序列
- 适用于：连续情节，章节紧密相关

**方案B**：分解为章节事件，每章独立场景
- 适用于：章节相对独立，每章有自己的小目标

**方案C**：AI自行设计的混合方案
- 根据具体需求创造最适合的结构

#### 为什么采用AI自由决策？

**问题分析**：
- 硬性规则限制太多，无法应对复杂多变的叙事需求
- 不同类型的中型事件需要不同的处理方式
- 过度约束会降低创作的灵活性和质量

**解决方案**：
- 给AI充分的自由度，让它根据实际情况做出最佳决策
- 只保留"避免重复"这一个核心约束
- 让专业判断代替死板的规则

这个方案既保证了内容质量，又给予了创作足够的灵活性。

## 注意事项

1. **不要在第一阶段调用** `smart_decompose_medium_events()` 方法
2. **不要在第一阶段生成** `chapter_scene_events`
3. 第一阶段的验证应该宽松，未覆盖章节不是错误
4. 所有修改都应该保留向后兼容性，以便第二阶段可以基于第一阶段的结果继续工作
