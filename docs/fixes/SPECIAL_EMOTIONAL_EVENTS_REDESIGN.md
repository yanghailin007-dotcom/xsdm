# 特殊情感事件重新设计方案

## 问题描述

### 原始问题
用户发现特殊情感事件与中级事件存在章节重叠问题：
- 中级事件的 `chapter_range` 已经排满了整个重大事件范围
- 特殊情感事件也有 `chapter_range`，导致章节重叠
- 在UI中显示时造成混乱，两种事件都占用相同的章节

### 根本原因
从 [`event_decomposer.py`](../src/managers/stage_plan/event_decomposer.py:196-198) 的prompt中发现设计矛盾：

```python
4.  【绝对覆盖指令】: 你生成的所有中型事件和特殊情感事件的chapter_range，必须完整且无缝地覆盖父级"重大事件"的整个章节范围。
```

这个指令在逻辑上是不可能实现的，因为中级事件已经覆盖了所有章节。

## 解决方案

### 核心设计思想
**特殊情感事件不是独立的章节事件，而是附着在中型事件上的情感元素**

### 数据结构重新设计

#### 第一阶段（事件分解）
特殊情感事件现在作为中型事件的子字段：

```python
{
    "name": "中型事件A",
    "chapter_range": "10-15",
    "special_emotional_events": [
        {
            "name": "情感互动1",
            "target_chapter": 12,  # 只指定章节，不指定范围
            "purpose": "深化角色关系",
            "emotional_tone": "温馨",
            "key_elements": ["对话", "眼神交流", "肢体语言"],
            "context_hint": "在中型事件的转折点"  # 可选
        }
    ]
}
```

#### 第二阶段（场景生成）
在生成场景时，将特殊情感事件作为**背景要求**提供给AI：

```
请为第12章设计场景事件。

本章的情感氛围要求：
- 需要融入【情感互动1】：深化角色关系
- 情感基调：温馨
- 关键元素：对话、眼神交流、肢体语言

请将这些情感元素自然地融入到场景中，让情感发展与情节推进有机结合起来。
```

**关键**：让AI根据情节需要，**自然地**将情感元素融入场景，而不是强制插入到某个位置。

### 优点
1. ✅ **避免章节重叠**：特殊情感事件不占用独立章节
2. ✅ **自然流畅**：AI自主决定如何融合，保持情节连贯
3. ✅ **灵活多样**：同一个情感事件可以有不同的融合方式
4. ✅ **简单清晰**：数据结构更简单，只指定章节即可
5. ✅ **避免割裂**：情感元素成为场景的有机组成部分

## 实施的修改

### 1. [`event_decomposer.py`](../src/managers/stage_plan/event_decomposer.py)

#### 修改1：更新事件分解prompt（第178-252行）
- 移除了对特殊情感事件的"绝对覆盖指令"
- 明确特殊情感事件应该作为中型事件的子字段
- 要求AI指定 `target_chapter` 而非 `chapter_range`
- 强调自然融合而非插入

#### 修改2：章节分解时融合特殊情感事件（第264-310行）
- 在 `_decompose_to_chapter_then_scene` 方法中
- 收集中型事件的特殊情感事件
- 按章节分组并添加到prompt中
- 强调"自然融合"而非"插入"

#### 修改3：多章场景构建时融合特殊情感事件（第401-454行）
- 在 `_decompose_direct_to_scene` 方法中
- 同样收集并融合特殊情感事件
- 筛选出在中型事件章节范围内的特殊情感事件

#### 修改4：单章场景构建时融合特殊情感事件（第637-723行）
- 在 `_decompose_single_chapter_with_complete_arc` 方法中
- 只包含目标章节为当前章节的特殊情感事件
- 提供详细的情感融合指导

### 2. [`scene_assembler.py`](../src/managers/stage_plan/scene_assembler.py)

#### 修改：移除顶层特殊情感事件列表（第40-125行）
- 不再将特殊情感事件提取到顶层的 `special_emotional_events` 列表
- 特殊情感事件保留在中型事件的 `special_emotional_events` 字段中
- 统计时遍历所有中型事件来收集特殊情感事件数量
- 更新了 `overall_approach` 说明，明确特殊情感事件附着在中级事件上

## 数据流变化

### 修改前
```
重大事件
├── composition（中型事件）
└── special_emotional_events（特殊情感事件，有chapter_range）
```

### 修改后
```
重大事件
└── composition（中型事件）
    └── special_emotional_events（特殊情感事件，只有target_chapter）
```

## 后续工作

### 前端显示逻辑
需要更新前端显示，将特殊情感事件作为中型事件的子元素展示：
1. 在故事线UI中，中型事件可展开显示其附着的特殊情感事件
2. 显示特殊情感事件的 `target_chapter`、`purpose`、`emotional_tone` 等信息
3. 不再将特殊情感事件作为独立的章节级事件显示

### 测试验证
1. 生成新的项目，验证第一阶段输出格式正确
2. 检查特殊情感事件是否正确附着到中型事件
3. 验证第二阶段场景生成时是否正确融合特殊情感事件
4. 确认没有章节重叠问题

## 影响范围

### 受影响的模块
- ✅ [`event_decomposer.py`](../src/managers/stage_plan/event_decomposer.py) - 事件分解器
- ✅ [`scene_assembler.py`](../src/managers/stage_plan/scene_assembler.py) - 场景组装器
- ⏳ 前端显示逻辑 - 需要更新
- ⏳ 现有项目数据 - 需要迁移脚本

### 兼容性
- 新生成的项目将使用新格式
- 旧项目可能需要数据迁移脚本
- 建议在生成新项目时使用新格式，旧项目保持不变

## 总结

这次重新设计解决了特殊情感事件与中级事件的章节重叠问题，通过将特殊情感事件作为中型事件的子元素，并在场景生成时自然融合，实现了：

1. **逻辑一致性**：不再有不可能实现的"绝对覆盖指令"
2. **数据清晰性**：特殊情感事件的归属明确
3. **情节流畅性**：情感元素自然融入，不生硬插入
4. **UI直观性**：层级关系清晰，易于理解

这是一个从根本上解决问题的方案，而不是打补丁式的修复。
